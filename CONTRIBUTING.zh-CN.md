# 贡献指南

我是 Alyce。如果您准备给这个仓库提改动，我会更希望整个过程保持清楚、可审查，也尽量别乱成一团。

## 基本约定

- 尽量让一次改动只服务于一个明确目标
- 不要提交 `.env`、`.alyce/` 或生成产物 `dist/`
- 如果修改运行时或工具层代码，请保留审批边界、文件范围检查和回滚能力
- 只要新增了用户可见设置或环境变量，就同步更新公开文档

## 本地准备

1. 安装依赖：

```bash
npm install
```

2. 用模板创建 `.env`：

```bash
copy .env.example .env
# 或者：cp .env.example .env
```

3. 至少填写：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `OPENAI_MODEL`

## 工作方式

- 运行时代码在 `src/`
- 对外文档在 `README.md` 和 `docs/`
- `User_Info/` 按用户私有资料处理，不视为仓库说明文档
- 代码风格请贴近现有 TypeScript：2 空格缩进、分号、双引号、相对导入带显式 `.js` 扩展名

## 验证要求

提交改动前，至少执行：

```bash
npm run build
```

如果改动涉及交互行为，还请在真实 TTY 里补一遍：

```bash
npm run dev
```

这些地方尤其值得认真检查：

- prompt 组装
- slash 命令
- 设置持久化
- 工具审批流程
- 记忆持久化与上下文预览

## Pull Request 建议

为了让 review 更顺一点，建议：

- 标题尽量短，用祈使句
- 简洁说明用户侧或运行时侧的变化
- 列出受影响的模块
- 写清楚自己实际跑过哪些验证
- 如果终端 UI 有明显变化，附上截图

## 文档改动约定

这个仓库的公开文档按“双语成对”维护：

- 英文文件作为默认入口
- 简体中文放在对应的 `*.zh-CN.md`
- 新增公开文档时，请同时补两种语言，并在顶部互相链接

GitHub 上看起来像“语言切换”的效果，本质上就是这一套纯 Markdown 的手工交叉链接，不是什么特殊功能。我……还是觉得写清楚比较安心。
