# 贡献指南

我是 Alyce。*写贡献指南的时候我总会有点紧张——万一看上去太苛刻了怎么办？但说实话，有几点共识对大家都轻松。*

如果你想给 Alyce-Agent 贡献代码，下面这些是我希望能记住的事。没什么为难人的——就足以让代码库保持可审查、历史记录保持干净。

## 基本约定

- **一次改动，一个目标。** 一个 PR 修了 bug、又重构了三个模块、又加了个新功能，那是三个 PR 披了一件风衣。别。
- **别交 `.env`、`.alyce/` 或 `dist/`。** 这些要么是运行时产物要么是秘密——不归 git 管。
- **保护好安全护栏。** 修改运行时或工具代码时，请保持审批边界、文件范围检查和回滚能力完好。它们存在是有原因的。
- **新设置有文档。** 加了用户可见的设置或环境变量，就同步更新公开文档。一个没人知道的设置等于不存在。

## 搭环境

```bash
npm install              # 拉依赖
copy .env.example .env   # Windows（Unix 用 cp）
# 然后填 OPENAI_API_KEY、OPENAI_BASE_URL、OPENAI_MODEL
```

就这些。可以编译和跑了。

## 代码风格

贴近现有的：

- 2 空格缩进
- 分号
- 双引号
- 相对导入显式带 `.js` 后缀
- 组件和工具类用 `PascalCase`，工具函数用 `camelCase`

*目前还没配 linter。我知道，我知道——但暂时就看看周围代码怎么写，跟着来。*

## 提交前检查

至少跑：

```bash
npm run build
```

它会全量编译 TypeScript 代码库。过了就说明结构上没问题。

如果你动了交互行为——命令、工具、提示词、UI——还请在真终端里试一遍：

```bash
npm run dev
```

值得手动检查的地方：

- **提示词组装** — `/context` 显示的是你预期的吗？
- **Slash 命令** — 还管用吗？
- **设置持久化** — 改了重启还在吗？
- **工具审批** — 该弹审批的时候弹了吗？
- **记忆** — `/remember` 记住了、`/memory` 能看见吗？

## Pull Request

让它好扫：

- **标题**：短，祈使句。比如"修复工具超时处理"而不是"修了一个工具的 bug"。
- **摘要**：从用户或运行时角度看，变了什么。
- **影响范围**：涉及哪些模块。
- **验证**：你实际测了什么，不是你打算测什么。
- **截图**：如果终端 UI 有可见变化就附上。

## 文档约定

这个仓库的公开文档按双语维护：

- 英文文件是 `docs/` 下的默认入口。
- 简体中文翻译统一在 `docs/zh-CN/`。
- README 的中文翻译在 `.github/readme-zh_cn.md`。
- 新增文档时，中英两版都要有，并在顶部互相链接。

GitHub 上看起来像"语言切换"的效果，就是纯 Markdown 的手工交叉链接。没魔法、没特殊 GitHub 功能——就是一个稳定一致的模式。
