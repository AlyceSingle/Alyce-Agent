# 安全说明

我是 Alyce。*这一页读起来会有点像文件规定。但当一个程序能在真实机器上读文件、跑命令、持久化记忆的时候，我觉得严谨一点是应该的。*

如果你在 Alyce-Agent 里发现了安全问题，请把它当安全漏洞来对待，而不是普通 bug。运行时对真实文件系统有真实访问权限——这意味着漏洞的影响范围不是理论上的。

## 什么算安全问题

以下这类问题应该按安全报告处理，不要当成普通 bug：

- **绕过审批关卡** — 本该弹审批却跳过去了。
- **突破文件范围限制** — 助手读写了允许目录以外的文件。
- **非预期写入或不安全回滚** — 文件快照静默失败，或回滚损坏了状态。
- **密钥泄露** — API key、token 或敏感内容泄露进了 prompt、日志或持久化状态。
- **命令执行越界** — shell 命令做的事超出了文档承诺的范围。
- **跨越信任边界的提示词注入** — 不可信内容能改写 system prompt 或覆盖安全规则。

*如果不确定够不够"严重"，宁愿按安全问题报。我宁可收到一个后来发现不严重的报告，也不想错过一个真的严重的。*

## 怎么报告

这个仓库目前还没有在仓库内公布私密安全联系方式。在正式渠道出来之前，请这样处理：

1. **先开一个最小化的公开 issue**，说明你有一个安全报告、需要私下沟通渠道。
2. **第一条公开说明里别放利用细节** — 不贴 payload、不贴复现步骤、不贴敏感数据。
3. **等维护者确认私密沟通方式后**，再共享完整细节。

*如果你就是维护者读到这儿：请把这一段换成真实的安全邮箱，或者开 GitHub Security Advisories。眼下这个过渡方案比啥都没有强，但确实不理想。*

## 什么信息会让报告更有用

你能提供越多这些内容，修复就越快：

- 受影响版本或 commit hash
- 操作系统和 shell
- 问题在 `npm run dev` 还是 `npm start` 里复现
- 涉及哪个功能区域
- 最小复现步骤
- 预期行为与实际行为
- 你的影响评估

## 响应预期

安全审查是尽力而为。没有正式 SLA——这是开源项目，不是托管服务。

不过，描述清楚、边界明确、能复现的报告，排查优先级会高很多。含糊不清、没有复现步骤的报告，往往只能在队列里等着我们猜你说的到底是什么。

## 负责任披露

以让 Alyce-Agent 更安全为目的的善意研究，非常欢迎。但请：

- 修复之前，别公开完整利用链。
- 别碰你没权限接触的第三方数据。
- 别对你不拥有、不控制的系统做破坏性测试。
