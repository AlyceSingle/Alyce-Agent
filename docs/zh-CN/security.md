# 安全说明

我是 Alyce。这个项目可以读文件、执行工具、组装 prompt、持久化记忆，所以安全问题最好还是比普通 bug 更谨慎一点处理。

## 适合报告的问题

如果涉及这些方向，请按安全问题对待：

- 绕过审批流程
- 越过路径访问范围
- 非预期文件写入或不安全的回滚行为
- 通过 prompt、日志或持久化状态泄露敏感信息
- 命令执行突破文档承诺的限制
- 跨信任边界传播的 prompt injection 路径

## 报告方式建议

这个仓库目前还没有在仓库内公开专门的私密安全联络方式。

在正式提供之前，请尽量不要在首次公开 issue 里直接贴完整利用细节。更稳妥一点的做法是：

1. 先开一个最小化 issue，说明这是一个需要私下沟通的安全报告
2. 第一条公开说明里不要放可直接利用的 payload 或敏感数据
3. 等建立私密沟通渠道后，再共享完整复现信息

如果您是维护者，我会建议把这里替换成真实可用的安全邮箱，或者 GitHub Security Advisory 流程。

## 什么信息会有帮助

比较有价值的内容包括：

- 受影响版本或 commit
- 操作系统与 shell
- 问题发生在 `npm run dev` 还是 `npm start`
- 涉及的功能区域
- 最小复现步骤
- 预期行为与实际行为
- 影响评估

## 响应预期

当前安全问题处理是尽力而为，没有正式 SLA。

不过，只要报告本身清楚、边界明确、可复现，排查起来通常会稳很多，也不容易误判。

## 负责任披露

欢迎以善意研究为目的的安全反馈。但也请尽量避免：

- 在修复前公开完整利用链
- 不必要地接触第三方数据
- 对您不拥有或无法控制的系统做破坏性测试

这部分看起来会有点正式。我还是宁愿把规则写清楚，毕竟这个仓库本来就能在本地机器上实际做事。
